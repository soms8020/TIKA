# Feature Specification: 티켓 생성 (Create Ticket)

**Feature Branch**: `001-create-ticket`

**Created**: 2026-06-15

**Status**: Draft

**Input**: User description: "API_SPEC.md의 POST /api/tickets 명세를 확인하고 구현에 필요한 요구사항을 정리해줘."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 최소 정보로 빠르게 티켓 생성 (Priority: P1)

사용자가 할 일이 떠올랐을 때, 제목만 입력해도 즉시 새 티켓이 만들어져 Backlog 칼럼 맨 위에 나타난다. 우선순위나 날짜 같은 부가 정보는 나중에 채울 수 있어야 한다.

**Why this priority**: 티켓 생성은 칸반 보드의 가장 기본이 되는 진입 행위다. 이것 없이는 보드에 담을 데이터 자체가 존재하지 않으므로 제품의 핵심 가치를 단독으로 전달하는 최소 기능이다.

**Independent Test**: 제목만 담아 생성 요청을 보냈을 때, Backlog 상태의 새 티켓이 기본 우선순위(MEDIUM)로 생성되고 생성 결과가 반환되는지로 단독 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 유효한 제목만 입력했을 때, **When** 티켓 생성을 요청하면, **Then** 상태 `BACKLOG`, 우선순위 `MEDIUM`인 새 티켓이 생성되고 생성된 전체 데이터(자동 부여된 id, position, 생성/수정 시각 포함)가 반환된다.
2. **Given** 제목·설명·우선순위·시작예정일·종료예정일을 모두 입력했을 때, **When** 티켓 생성을 요청하면, **Then** 입력한 값이 모두 반영된 티켓이 생성된다.
3. **Given** Backlog에 이미 티켓이 존재할 때, **When** 새 티켓을 생성하면, **Then** 새 티켓이 Backlog 칼럼의 맨 위(가장 작은 순서값)에 배치된다.

---

### User Story 2 - 잘못된 입력에 대한 명확한 피드백 (Priority: P2)

사용자가 제목을 비우거나 글자 수 제한을 초과하거나 잘못된 우선순위/과거 종료예정일을 입력하면, 티켓이 만들어지지 않고 무엇이 잘못됐는지 알 수 있는 메시지를 받는다.

**Why this priority**: 데이터 무결성을 보장하고 사용자가 실수를 즉시 교정하도록 돕는다. 생성 기능(P1)이 동작한 뒤 신뢰성을 확보하는 단계다.

**Independent Test**: 제목 없이 생성을 요청했을 때 생성이 거부되고 검증 실패 응답이 돌아오는지로 단독 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 제목이 비어 있거나 공백만 입력했을 때, **When** 생성을 요청하면, **Then** 생성이 거부되고 "제목을 입력해주세요" 메시지를 포함한 검증 실패 응답이 반환된다.
2. **Given** 제목이 200자를 초과할 때, **When** 생성을 요청하면, **Then** 생성이 거부되고 "제목은 200자 이내로 입력해주세요" 메시지가 반환된다.
3. **Given** 설명이 1000자를 초과할 때, **When** 생성을 요청하면, **Then** 생성이 거부되고 "설명은 1000자 이내로 입력해주세요" 메시지가 반환된다.
4. **Given** 우선순위가 LOW/MEDIUM/HIGH 외의 값일 때, **When** 생성을 요청하면, **Then** 생성이 거부되고 "우선순위는 LOW, MEDIUM, HIGH 중 선택해주세요" 메시지가 반환된다.
5. **Given** 종료예정일이 오늘보다 과거일 때, **When** 생성을 요청하면, **Then** 생성이 거부되고 "종료예정일은 오늘 이후 날짜를 선택해주세요" 메시지가 반환된다.

---

### Edge Cases

- **제목 앞뒤 공백**: 앞뒤 공백을 제거한 실제 내용이 1자 이상이어야 유효하다. 공백만으로 이루어진 제목은 빈 제목과 동일하게 거부한다.
- **빈 Backlog에 첫 티켓 생성**: 칼럼에 기존 티켓이 없을 때도 순서값이 일관되게 부여되어야 한다(가정: 첫 티켓의 순서값은 0).
- **종료예정일 = 오늘**: 오늘 날짜는 허용한다("오늘 이후"는 오늘 포함).
- **시작예정일과 종료예정일의 선후 관계**: 본 명세(API_SPEC.md)는 두 날짜 간 선후 관계를 검증하지 않는다. 검증 대상이 아니다.
- **설명/날짜 미입력**: 선택 필드는 미입력 시 null로 저장된다.
- **명세에 정의되지 않은 추가 필드 전송**: 검증 스키마에 정의된 필드만 사용하며 그 외 필드는 무시한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 제목만으로 새 티켓을 생성할 수 있어야 한다(설명, 우선순위, 시작예정일, 종료예정일은 선택).
- **FR-002**: 시스템은 새로 생성된 티켓의 상태를 항상 `BACKLOG`로 설정해야 한다.
- **FR-003**: 시스템은 우선순위가 입력되지 않은 경우 기본값 `MEDIUM`을 적용해야 한다.
- **FR-004**: 시스템은 새 티켓을 Backlog 칼럼의 맨 위(기존 티켓들의 최소 순서값보다 1024 작은 값)에 배치해야 한다.
- **FR-005**: 시스템은 생성 시 생성 시각과 수정 시각을 자동으로 기록해야 한다.
- **FR-006**: 시스템은 생성된 티켓의 전체 데이터(id, title, description, status, priority, position, plannedStartDate, dueDate, startedAt, completedAt, createdAt, updatedAt)를 반환해야 한다. 새로 생성된 티켓의 startedAt·completedAt은 null이다.
- **FR-007**: 시스템은 제목을 필수로 요구하며, 비어 있거나 공백만 있는 제목을 거부해야 한다.
- **FR-008**: 시스템은 제목 길이를 1~200자로 제한해야 한다.
- **FR-009**: 시스템은 설명 길이를 최대 1000자로 제한해야 한다.
- **FR-010**: 시스템은 우선순위를 LOW, MEDIUM, HIGH 중 하나로만 허용해야 한다.
- **FR-011**: 시스템은 시작예정일과 종료예정일을 YYYY-MM-DD 형식으로만 허용해야 한다.
- **FR-012**: 시스템은 종료예정일이 오늘(포함) 이후 날짜인지 검증하고, 과거 날짜를 거부해야 한다.
- **FR-013**: 검증에 실패하면 시스템은 티켓을 생성하지 않고, 통일된 에러 형식(`{ error: { code, message } }`, code=`VALIDATION_ERROR`)으로 명세에 정의된 한국어 메시지를 반환해야 한다.
- **FR-014**: 생성 성공 시 시스템은 "생성됨"을 의미하는 응답으로 결과를 반환하고, 검증 실패 시 "잘못된 요청"을 의미하는 응답을 반환해야 한다(구체 상태 코드/형식은 API_SPEC.md를 따른다).

### Key Entities *(include if feature involves data)*

- **Ticket(티켓)**: 하나의 할 일 항목. 사용자 입력 속성(제목, 설명, 우선순위, 시작예정일, 종료예정일)과 시스템 관리 속성(고유 식별자, 상태, 칼럼 내 순서, 시작일, 종료일, 생성 시각, 수정 시각)을 가진다. 생성 시점에는 항상 Backlog 칼럼에 속한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자는 제목 하나만 입력하여 단일 동작으로 새 티켓을 생성할 수 있다.
- **SC-002**: 유효한 입력으로 생성한 티켓은 100% Backlog 칼럼 맨 위에 나타난다.
- **SC-003**: 명세에 정의된 6가지 검증 실패 케이스(제목 누락/공백, 제목 초과, 설명 초과, 잘못된 우선순위, 과거 종료예정일)는 모두 티켓을 생성하지 않고 해당하는 안내 메시지를 반환한다.
- **SC-004**: 생성된 티켓 응답은 명세에 정의된 12개 필드를 빠짐없이 포함하며, 우선순위 미입력 시 MEDIUM, 상태는 BACKLOG로 일관되게 채워진다.

## Assumptions

- **빈 칼럼의 순서값**: Backlog에 기존 티켓이 없을 때 새 티켓의 순서값은 0으로 한다(DATA_MODEL.md의 칼럼별 순서 시작값 0 관례 기준).
- **"오늘"의 기준**: 종료예정일 검증의 "오늘"은 서버 기준 날짜로 판정한다(API_SPEC.md의 Zod 스키마가 서버 현재 날짜를 사용).
- **시작예정일에는 과거 제약이 없다**: 종료예정일만 "오늘 이후" 제약을 가지며, 시작예정일은 과거여도 허용된다(API_SPEC.md 기준).
- **인증 없음**: MVP 단일 사용자 가정으로 생성 시 인증/권한 검사는 수행하지 않는다(API_SPEC.md 기준).
- **부분 입력 오류의 결합**: 여러 필드가 동시에 잘못된 경우의 메시지 우선순위는 명세에 정의되지 않았으며, plan 단계에서 검증 라이브러리의 기본 동작을 따른다.
- **단일 진실 공급원**: 요청/응답 형식, 검증 규칙, 에러 코드·메시지는 `docs/API_SPEC.md`와 `docs/DATA_MODEL.md`를 따른다. 본 spec과 충돌 시 해당 명세를 먼저 정정한다.

## Dependencies

- 본 기능은 GET /api/tickets(보드 조회)와 독립적으로 생성·테스트·배포 가능하나, 생성 결과를 화면에서 확인하려면 보드 조회 기능이 필요하다.
- 데이터 저장소(tickets 테이블)와 공유 검증 스키마(`src/shared/validations`)에 의존한다.
