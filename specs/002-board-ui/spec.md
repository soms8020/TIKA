## User Scenarios & Testing *(mandatory)*

### User Story 1 - 칸반 보드 현황 파악 (Priority: P1)

사용자가 앱을 열면 모든 할 일이 상태별 칼럼(Backlog 사이드바 + TODO / In Progress / Done)에 분류되어 표시된다. 각 칼럼은 카드 수를 보여주고, 카드는 정해진 순서대로 정렬된다. 각 카드에는 제목, 설명, 우선순위, 종료예정일이 표시된다.

**Why this priority**: 보드 조회는 다른 모든 기능(생성·이동·수정·삭제·필터)이 올라가는 토대다. 이것만 있어도 "현재 업무 현황을 한눈에 본다"는 핵심 가치를 단독으로 전달한다.

**Independent Test**: 시드 데이터가 있는 상태에서 페이지를 열어 4개 칼럼, 칼럼별 카드 수, position 순서, 카드 내용(제목/설명/우선순위/종료예정일)이 정확히 표시되는지로 단독 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 여러 칼럼에 티켓이 존재하는 상태, **When** 보드를 연다, **Then** Backlog는 좌측 사이드바, TODO/In Progress/Done은 우측 3칼럼에 배치되어 표시된다.
2. **Given** 한 칼럼에 여러 티켓이 있는 상태, **When** 칼럼을 본다, **Then** 카드가 position 오름차순으로 정렬되고 칼럼 헤더에 카드 수가 표시된다.
3. **Given** 티켓이 없는 칼럼, **When** 그 칼럼을 본다, **Then** "이 칼럼에 티켓이 없습니다" 안내가 표시된다.
4. **Given** 우선순위가 다른 티켓들, **When** 카드를 본다, **Then** LOW 회색 / MEDIUM 파랑 / HIGH 빨강 뱃지로 구분되어 보인다.

### User Story 2 - 새 할 일 등록 (Priority: P1)

사용자가 "새 업무" 버튼으로 생성 폼을 열고, 제목(필수)과 선택 항목(설명·우선순위·시작예정일·종료예정일)을 입력해 새 티켓을 만든다. 생성된 티켓은 Backlog 맨 위에 즉시 나타난다.

**Why this priority**: 보드에 데이터를 넣는 유일한 사용자 경로다. 조회(US1)와 함께 있어야 "기록하고 본다"는 최소 제품이 성립한다.

**Independent Test**: 빈 보드에서 "새 업무"로 제목만 입력해 생성하면 Backlog 맨 위에 카드가 추가되고 폼이 닫히는지로 단독 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 보드 화면, **When** "새 업무" 버튼을 클릭한다, **Then** 생성 폼(모달)이 열린다.
2. **Given** 생성 폼, **When** 제목만 입력하고 제출한다, **Then** 우선순위 MEDIUM이 기본 적용된 카드가 Backlog 맨 위에 추가되고 폼이 닫힌다.
3. **Given** 생성 폼, **When** 제목을 비우고 제출한다, **Then** "제목을 입력해주세요" 에러가 인라인 표시되고 제출되지 않는다.
4. **Given** 생성 폼, **When** 종료예정일에 과거 날짜를 선택한다, **Then** "종료예정일은 오늘 이후 날짜를 선택해주세요" 에러가 표시된다.

### User Story 3 - 드래그앤드롭 상태/순서 변경 (Priority: P2)

사용자가 카드를 드래그하여 다른 칼럼으로 옮기거나 같은 칼럼 내 순서를 바꾼다. 화면은 즉시 반영(낙관적 업데이트)되고, 저장 실패 시 원래 위치로 되돌아간다. Done으로 옮기면 완료 처리된다.

**Why this priority**: 칸반의 핵심 상호작용이지만, 조회·생성이 선행되어야 의미가 있다.

**Independent Test**: 카드를 다른 칼럼/위치로 드롭했을 때 즉시 이동하고, 저장 실패를 모사하면 원위치로 롤백되는지로 단독 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** Backlog의 카드, **When** TODO 칼럼으로 드롭한다, **Then** 카드가 TODO로 즉시 이동하고 시작일이 기록된다.
2. **Given** TODO의 카드, **When** Backlog로 되돌려 드롭한다, **Then** 카드가 Backlog로 이동하고 시작일이 초기화된다.
3. **Given** 임의 칼럼의 카드, **When** Done 칼럼으로 드롭한다, **Then** 완료 처리되어 종료일이 기록된다.
4. **Given** 같은 칼럼 내 카드, **When** 위/아래로 드래그한다, **Then** 칼럼 내 순서가 변경된다.
5. **Given** 카드 이동 중 저장이 실패하는 상황, **When** 드롭한다, **Then** 화면이 이동 전 상태로 롤백되고 에러가 안내된다.

---

### User Story 4 - 할 일 수정 (Priority: P2)

사용자가 카드를 클릭해 상세 모달을 열고, 제목·설명·우선순위·시작예정일·종료예정일을 수정한다. 상태·시작일·종료일·생성일은 읽기 전용으로 표시된다.

**Why this priority**: 등록한 할 일의 정보를 최신으로 유지하는 보조 흐름. 조회/생성 이후 가치가 있다.

**Independent Test**: 카드 클릭 → 모달에서 제목 변경 → 저장 시 보드 카드가 갱신되는지로 단독 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 보드의 카드, **When** 카드를 클릭한다, **Then** 상세 모달이 열리고 편집 가능 필드와 읽기 전용 필드(상태·시작일·종료일·생성일)가 구분 표시된다.
2. **Given** 상세 모달, **When** 편집 필드를 수정하고 저장한다, **Then** 변경 사항이 보드 카드에 즉시 반영된다.
3. **Given** 상세 모달, **When** 검증에 어긋난 값을 입력한다, **Then** 해당 필드에 에러 메시지가 인라인 표시된다.
4. **Given** 열린 모달, **When** 바깥 영역 클릭 또는 ESC를 누른다, **Then** 모달이 닫힌다.

---

### User Story 5 - 할 일 삭제 (Priority: P3)

사용자가 상세 모달에서 삭제를 선택하면 확인 다이얼로그가 뜨고, 확인 시 티켓이 영구 삭제되어 보드에서 사라진다.

**Why this priority**: 정리용 보조 기능. 핵심 흐름 이후에 추가되어도 무방하다.

**Independent Test**: 모달에서 삭제 → 확인 → 카드가 보드에서 제거되고 모달이 닫히는지로 단독 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 상세 모달, **When** 삭제 버튼을 클릭한다, **Then** "정말 삭제하시겠습니까?" 확인 다이얼로그가 표시된다.
2. **Given** 확인 다이얼로그, **When** 확인을 누른다, **Then** 티켓이 삭제되어 보드에서 사라지고 모달이 닫힌다.
3. **Given** 확인 다이얼로그, **When** 취소를 누른다, **Then** 아무 변화 없이 다이얼로그만 닫힌다.

---

### User Story 6 - 마감 초과 인지 (Priority: P2)

종료예정일이 지난 미완료 티켓(Done 제외)은 다른 카드와 시각적으로 명확히 구분되는 오버듀 표시가 나타난다.

**Why this priority**: 지연 업무 인지라는 뚜렷한 사용자 가치. 조회 위에 얹는 표시 계층이라 P2.

**Independent Test**: 종료예정일이 과거인 미완료 티켓에 오버듀 표시가 나타나고, Done이거나 미래 기한인 티켓에는 나타나지 않는지로 단독 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 종료예정일이 과거이고 상태가 Done이 아닌 카드, **When** 보드를 본다, **Then** 오버듀 표시(예: 빨간 테두리)가 나타난다.
2. **Given** 종료예정일이 과거이지만 Done 상태인 카드, **When** 보드를 본다, **Then** 오버듀 표시가 나타나지 않는다.

---

### User Story 7 - 필터 (이번주 업무 / 일정 초과) (Priority: P3)

사용자가 메인 영역 상단의 필터 버튼으로 "이번주 업무" 또는 "일정 초과" 티켓만 보거나, 다시 눌러 전체 보기로 돌아간다. 필터는 Backlog에는 적용되지 않는다.

**Why this priority**: 편의 기능. 보드 조회·표시가 완성된 뒤에 가치가 있다.

**Independent Test**: 필터 클릭 시 해당 조건의 카드만 메인 칼럼에 남고, 재클릭 시 전체로 복귀하며 Backlog는 항상 전체 표시되는지로 단독 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 다양한 기한의 티켓, **When** "이번주 업무" 필터를 켠다, **Then** 이번 주 종료예정일을 가진 TODO/In Progress 티켓만 메인 칼럼에 표시된다.
2. **Given** 오버듀 티켓이 섞인 보드, **When** "일정 초과" 필터를 켠다, **Then** 오버듀 티켓만 메인 칼럼에 표시된다.
3. **Given** 활성화된 필터, **When** 같은 필터 버튼을 다시 클릭한다, **Then** 전체 보기로 돌아간다.
4. **Given** 임의의 필터가 켜진 상태, **When** Backlog를 본다, **Then** Backlog는 필터와 무관하게 전체 티켓을 표시한다.

---

### Edge Cases

- 보드에 티켓이 하나도 없을 때: 모든 칼럼이 빈 상태 안내를 표시한다.
- 드래그를 시작했다가 같은 위치(원래 칼럼·원래 자리)에 그대로 놓을 때: 변경/저장 호출 없이 무시한다.
- 저장 요청 실패(네트워크/서버 오류): 낙관적 변경을 롤백하고 사용자에게 오류를 안내한다.
- Done 칼럼: 완료 후 일정 시간(24시간)이 지난 티켓은 보드에 표시되지 않는다(서버 필터링 결과를 그대로 반영).
- 매우 긴 제목/설명: 카드에서 말줄임 처리하여 레이아웃이 깨지지 않는다.
- 화면 폭 변화: 데스크톱(3칼럼)/태블릿(2칼럼)/모바일(단일 칼럼) 반응형으로 배치가 바뀐다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 서버에서 받은 보드 데이터를 Backlog(사이드바) + TODO / In Progress / Done(메인 3칼럼) 구조로 표시해야 한다.
- **FR-002**: 시스템은 각 칼럼의 카드를 position 오름차순으로 정렬하고, 칼럼 헤더에 카드 수를 표시해야 한다.
- **FR-003**: 시스템은 카드에 제목, 설명, 우선순위 뱃지(LOW 회색/MEDIUM 파랑/HIGH 빨강), 종료예정일(있을 경우)을 표시해야 한다.
- **FR-004**: 시스템은 종료예정일이 지난 미완료(Done 아님) 티켓에 오버듀 표시를 해야 한다.
- **FR-005**: 사용자는 "새 업무"로 생성 폼을 열어 제목(필수)·설명·우선순위·시작예정일·종료예정일을 입력해 티켓을 생성할 수 있어야 하며, 생성된 티켓은 Backlog 맨 위에 추가되어야 한다.
- **FR-006**: 시스템은 생성/수정 폼 입력을 제출 전 검증하고, 위반 시 정해진 에러 메시지를 해당 필드에 인라인 표시해야 한다(제목 필수·200자, 설명 1000자, 종료예정일 오늘 이후 등).
- **FR-007**: 사용자는 카드를 드래그하여 다른 칼럼으로 이동하거나 같은 칼럼 내 순서를 변경할 수 있어야 한다(역방향 포함, 이동 제약 없음).
- **FR-008**: 시스템은 드래그앤드롭 변경을 화면에 즉시 반영(낙관적 업데이트)하고, 저장 실패 시 이전 상태로 롤백하며 오류를 안내해야 한다.
- **FR-009**: 시스템은 카드를 Done으로 이동하면 완료 처리(종료일 기록)하고, 그 외 칼럼(BACKLOG/TODO/In Progress) 이동은 상태·순서 변경으로 처리해야 한다.
- **FR-010**: 시스템은 TODO로 이동 시 시작일을 기록하고, TODO에서 BACKLOG로 되돌리면 시작일을 초기화해야 한다(서버 규칙을 화면에 반영).
- **FR-011**: 사용자는 카드를 클릭해 상세 모달을 열 수 있어야 하며, 모달은 편집 가능 필드(제목·설명·우선순위·시작예정일·종료예정일)와 읽기 전용 필드(상태·시작일·종료일·생성일)를 구분 표시해야 한다.
- **FR-012**: 사용자는 상세 모달에서 편집 필드를 수정·저장할 수 있어야 하고, 변경 사항은 보드 카드에 즉시 반영되어야 한다.
- **FR-013**: 사용자는 상세 모달에서 삭제를 선택할 수 있어야 하며, 삭제는 확인 다이얼로그 확인 후에만 실행되어 보드에서 카드를 제거해야 한다.
- **FR-014**: 사용자는 "이번주 업무"·"일정 초과" 필터를 켜고 끌 수 있어야 하며, 필터는 메인 칼럼에만 적용되고 Backlog에는 적용되지 않아야 한다. 활성 필터를 다시 누르면 전체 보기로 복귀해야 한다.
- **FR-015**: 모달은 바깥 영역 클릭 또는 ESC로 닫혀야 하고, 열려 있는 동안 배경 스크롤이 잠겨야 한다.
- **FR-016**: 시스템은 데스크톱(3칼럼)·태블릿(2칼럼)·모바일(단일 칼럼) 화면 폭에 따라 반응형으로 배치되어야 한다.
- **FR-017**: 시스템은 키보드 포커스·조작(카드 포커스 및 Enter로 상세 열기)과 스크린 리더용 레이블(role/aria-label)을 제공해야 한다.
- **FR-018**: 카드에 대한 드래그와 클릭(상세 열기)은 사용자 의도대로 구분되어야 한다(미세한 움직임은 클릭으로 처리).

### Key Entities *(include if feature involves data)*

- **티켓(Ticket)**: 할 일 한 건. 제목, 설명, 상태(Backlog/TODO/In Progress/Done), 우선순위(LOW/MEDIUM/HIGH), 칼럼 내 위치, 시작예정일, 종료예정일, 시작일, 종료일, 생성/수정 시각을 가진다. 종료예정일이 지났고 미완료면 "오버듀"로 파생 판정된다.
- **보드(Board)**: 4개 상태 칼럼별로 그룹화된 티켓 묶음. 화면 표시의 단위이며 사용자 조작(이동·생성·수정·삭제·필터)의 대상이다.
- **필터(Filter)**: 메인 칼럼 표시 범위를 좁히는 화면 상태. "전체 / 이번주 업무 / 일정 초과" 중 하나.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자는 보드를 열어 추가 조작 없이 모든 할 일의 상태별 분류를 한 화면에서 파악할 수 있다.
- **SC-002**: 사용자는 "새 업무" 클릭부터 생성 완료(카드가 Backlog에 표시)까지 30초 이내, 5회 이하의 조작으로 완료할 수 있다.
- **SC-003**: 카드를 드롭하면 화면 반영이 즉각적으로 보이며(체감 지연 없이), 저장 실패 시 100%의 경우 원래 위치로 정확히 복원된다.
- **SC-004**: 오버듀 티켓은 일반 티켓과 별도 안내 없이도 사용자가 즉시 구별할 수 있다(시각적 구분 100% 적용).
- **SC-005**: 데스크톱·태블릿·모바일 폭에서 가로 스크롤이나 레이아웃 깨짐 없이 보드를 사용할 수 있다.
- **SC-006**: 검증 위반 입력은 100% 제출 전에 차단되고, 사용자는 어떤 필드가 왜 잘못됐는지 메시지로 알 수 있다.
- **SC-007**: 필터를 켜고 끄는 동안 Backlog의 표시 내용은 영향을 받지 않는다.

## Assumptions

- 백엔드 API(`app/api`)와 응답 형식(API_SPEC.md), 공유 타입(`src/shared/types`), API 호출 모듈(`src/client/api/ticketApi.ts`), 핵심 스타일(`src/styles/globals.css`)은 이미 존재하며 그대로 활용한다.
- 보드 초기 데이터는 서버에서 렌더 시점에 제공된다(별도 로딩 화면 명세는 본 기능 범위 밖).
- 인증/사용자 구분은 MVP 범위 밖이다(단일 사용자 가정).
- 검색창(SearchInput)은 2차 구현 대상으로, 본 기능에서는 비활성 placeholder로만 렌더링한다.
- Done 칼럼의 24시간 표시 제한 등 시간 기반 필터링은 서버가 수행하며, 프론트엔드는 그 결과를 표시한다.
- "이번주" 범위는 해당 주 월요일~일요일로 본다.
- 오버듀 판정(종료예정일 < 오늘 AND 상태 ≠ Done)은 서버 파생 필드를 기본으로 하되, 프론트엔드에서도 동일 규칙으로 계산할 수 있다.
